Decisão Estratégica: PT Only, Multi-Market

Escopo restrito a Principal Tokens — yield fixo garantido até maturidade.

03

Separação de Responsabilidades V5.6

O princípio central da V5.6: agente decide SE, motor decide COMO / QUANDO / QUANTO.

Camada 1 — Agente (main.py)
Decide SE entrar num mercado
Resultado: HOLD / ROTATE_TO_PT / REBALANCE_TO_PT
  • Pre-Validate — Gates: TVL, DTM, APY mínimos
  • Risk Guard GRS — Score 4D de risco multidimensional
  • Claude regime — HIGH_YIELD / NEUTRAL / LOW_YIELD
  • ΔNet threshold — Comparação com limiar × bias
  • Salva decisão em pendle_recommendations
Camada 2 — Motor TDYM (negotiation_engine.py)
Decide COMO / QUANDO / QUANTO
Input: apenas REBALANCE_TO_PT aprovados
  • Calcula desvio — |portfolio_yield − 13.00%|
  • Escolhe estratégia — BUY direto vs SWAP de posição
  • Sizing — Tamanho que minimiza desvio do target
  • DCA — Parcelamento se volatilidade alta
  • Candidatos vêm exclusivamente da Camada 1
Armadilha crítica (corrigida): Conflating as duas camadas causou erros de implementação no início da V5.6. O motor de negociações não reavalia se o mercado é bom — isso já foi decidido pelo agente. O motor só decide a mecânica da execução.

Framework TDYM — Target Yield Deviation Minimization V5.6

Rejeita maximização cega de APY. Minimiza o desvio absoluto do portfólio em relação ao target de 13.00% anual.

Fórmulas centrais — Motor Quantitativo V5.6
// TDYM — Objetivo principal Minimize: |portfolio_yield − target_yield| where target_yield = 13.00% ← constante imutável // Decisão de entrada (herdada V5.5) ΔNet = (Fixed APY − Implied APY) − CTEA − Risco_Saída(0.5%) CTEA = (Gas_USD + Slip_entrada + Slip_saída) / Notional × (365 / DTM) Threshold = Δtarget_base(0.0080) + k_vol(2.0) × YVI(σ24h) // Scoring de Recomendação (V5.6) escore_final = (score_quant × 0.35) + (score_payload × 0.25) + (score_grs × 0.40) Decisão: ROTATE_TO_PT quando |ΔNet| > Threshold × regime_bias

Classificação de Risco — FRONTEND vs PROTOCOL

O Risk Guard distingue riscos operacionais de riscos on-chain para evitar falsos positivos.

Tipo Exemplos Severity Ação do Agente
FRONTEND_RISK Twitter comprometido, domain hijack, phishing, Discord hack LOW/MEDIUM NÃO bloqueia — opera via API/contratos
PROTOCOL_RISK Exploit de contrato, hack on-chain, depeg, governance attack, liquidity crisis HIGH/CRITICAL BLOQUEIA — circuit breaker automático
Incidente real (03/06/2026): Conta X/Twitter do Pendle comprometida via social engineering. Classificado como FRONTEND_RISK — agente continuou operando normalmente. API Pendle e contratos on-chain intactos. TVL $10.8M estável.

Global Risk Score (GRS) — 4 dimensões:

DimensãoPesoFonte
Liquidez de mercado 35% On-chain TVL, depth
Código do contrato (Slither) 30% Arbiscan API + whitelist auditados
Falhas operacionais 20% DB histórico pendle_risk_assessments
Reputacional externo 15% Perplexity sonar (4x/dia)
