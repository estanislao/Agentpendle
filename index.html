Decisão Estratégica: PT Only, Multi-Market

Escopo restrito a Principal Tokens (PT) — yield fixo garantido até maturidade. YT excluído para capital de usuários por incompatibilidade com a proposta DollarUp de rendimento passivo sem risco de perda do principal.

Atualização V5.5.1: O agente evoluiu de single-market (apenas PT-sUSDai-OCT2026) para multi-market. A lista de mercados monitorados é carregada dinamicamente do banco de dados PostgreSQL (pendle_monitored_markets), sem hardcode no código. Novos mercados são adicionados pelo dashboard sem reiniciar o agente.

Classificação de Risco Externo — FRONTEND vs PROTOCOL

O Perplexity Risk Scanner distingue dois tipos de incidente. Apenas riscos on-chain bloqueam o agente — riscos de frontend (comprometimento de conta social, phishing, domain hijack) não afetam operações via API/contratos.

TipoExemplosSeverityAção do Agente
FRONTEND_RISKTwitter comprometido, domain hijack, phishing, Discord hackLOW/MEDIUMNÃO bloqueia — opera via API/contratos
PROTOCOL_RISKExploit de contrato, hack on-chain, depeg, governance attack, liquidity crisisHIGH/CRITICALBLOQUEIA — circuit breaker automático
Incidente real detectado (03/06/2026): Conta X/Twitter do Pendle comprometida via social engineering. Classificado como FRONTEND_RISK — agente continuou operando normalmente. API Pendle e contratos on-chain intactos (TVL $10.8M estável).

Motor Quantitativo V5.5 — Lógica de Decisão

Fórmulas Centrais

ΔNet = (Fixed APY − Implied APY) − CTEA − Risco_Saída(0.5%)

CTEA = (Gas_USD + Slip_entrada + Slip_saída) / Notional × (365 / DTM)

Threshold = Δtarget_base(0.0080) + k_vol(2.0) × YVI(σ24h)

Decisão: ROTATE_TO_PT quando |ΔNet| > Threshold × regime_bias

Árvore de Decisão por Ciclo

1. Pre-validate (Gates): TVL < mínimo → skip | DTM < 15d → skip | APY < 3% → skip
2. Claude (cache): regime=LOW_YIELD + conf ≥ 65% → HOLD forçado, não entra em PT
3. Perplexity (cache): severity=HIGH/CRITICAL + PROTOCOL_RISK → bloqueia agente
4. Quant: calcula ΔNet, Threshold, CTEA → compara com limiar × bias
5. Decisão: ROTATE_TO_PT | HOLD_NO_OP | EXIT_PT → salva em pendle_recommendations
6. Aprovação humana: PENDING → operador aprova no dashboard → APPROVED → Safe executa

Fluxo de Aprovação e Execução

Modo atual: DRY_RUN. Nenhuma TX on-chain é executada. O botão EXEC no dashboard simula e loga a operação. Para ir LIVE, alterar PENDLE_EXECUTION_MODE=LIVE e reiniciar o agente.
1. Agente analisa → salva em pendle_recommendations com status=pending
2. Dashboard mostra → operador revisa ΔNet, Regime, Risco e clica EXEC (✓)
3. status=approved → agente detecta no próximo ciclo (até 300s)
4. safe_executor.py → constrói calldata Pendle Router V3, assina via Safe multisig 1-de-3
5. status=executed → TX hash registrado, AUM debitado, posição PT criada on-chain
