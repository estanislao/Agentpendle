Agente Inteligente de Alocação Pendle (V5.3 - Mission-Critical Deployment)

1. Objetivo do Sistema

O objetivo primário deste sistema é monitorar, predizer e arbitrar mercados do protocolo Pendle Finance na rede Arbitrum e ambientes Cross-Chain associados. O sistema gerencia o rebalanceamento dinâmico entre Yield Tokens (YT) e Principal Tokens (PT) usando inteligência preditiva acoplada para alternar regimes operacionais, mitigando sistematicamente riscos de cauda, slippage estrutural e custos de transporte invisíveis.

2. Modelo de Yield Nativos do Protocolo

O algoritmo de monitoramento rege as suas premissas exclusivamente pela arquitetura matemática do Pendle:

  • Underlying APY: Média móvel ponderada de 7 dias do rendimento nativo distribuído pelo ativo de lastro.
  • Implied APY: Projeção anualizada de yield futuro precificada instantaneamente pela curva da AMM do Pendle.
  • Fixed APY: Taxa de retorno garantida travada ao se adquirir o Principal Token (PT), agindo de forma análoga a um título de cupom zero.
  • Yield Token (YT): Ativo que concede direito integral aos fluxos de rendimento gerados pelo ativo subjacente até o vencimento.
  • Mecanismo de Claim e Expiração: Os yields acumulados pelo YT devem ser reivindicados de forma ativa via chamadas ao método claimYield(). Caso o resgate não ocorra antes da data de maturidade, todos os fluxos de caixa acumulados não reivindicados são permanentemente **zerados**, cessando os direitos contratuais do detentor.

3. Matriz de Ativos Parametrizável e Limites Globais

Nenhuma constante de negócio ou de mercado deve residir de forma rígida (hard-coded) no núcleo do agente. O sistema consome uma estrutura contextual dinâmica (AssetContext), permitindo alteração de limiares em tempo de execução via camadas de governança:

4. Lógica de Decisão e Consistência Dimensional de Custos

Para assegurar que o motor de decisão não autorize ordens cujas margens financeiras sejam absorvidas pelas taxas, as fricções operacionais são convertidas matematicamente para a métrica de Custos Totais Equivalentes Anuais (CTEA):

4.1. Equações Operacionais Expandidas

  1. Delta de Yield Bruto (ΔYield):
    ΔYield = Underlying APY7d − Implied APY
  2. Delta Ajustado ao Tempo (ΔTime): Onde tmaturidade = DaysToMaturity / 365.
    ΔTime = ΔYield × tmaturidade
  3. Custos Totais Equivalentes Anuais (CTEA):
    CTEA = (Gas% + Bridge% + SlippageIn + SlippageOut) / tmaturidade
  4. Delta Líquido Operacional Final (ΔNet):
    ΔNet = ΔTime − CTEA − RiscoSaída%

4.2. Formalização Matemática da Zona de Inércia Dinâmica (Δtarget_effective)

Para mitigar sinais falsos (*whipsawing*) em cenários de alta volatilidade, o limiar efetivo de trade expande e contrai de acordo com o desvio padrão amostral das últimas 24 horas:

Δtarget_effective = Δtarget_base + (k × σΔNet, 24h)

Onde Δtarget_base representa o spread mínimo estrutural (default: 50 bps ou 0.005) e k é o coeficiente multiplicador de sensibilidade ao risco (default: 1.5).

  • Ação YT: Ativada se ΔNet > Δtarget_effective.
  • Ação PT: Ativada se ΔNet < −Δtarget_effective.
  • No-Op (Hold): Mantido estritamente se |ΔNet| ≤ Δtarget_effective.
4.3. Alinhamento Dimensional do Breakeven para YT (YT Decay Gate)

Para evitar a entrada em posições de YT incapazes de gerar retorno real antes de expirarem, o cálculo do ponto de equilíbrio em dias (BreakevenDays) adota normalização decimal estendida e base temporal unificada (WAD / 18 casas decimais):

BreakevenDays = Preço\_YT / (Underlying\_APY7d / 365)

Se BreakevenDays > DaysToMaturity, a execução é **abortada imediatamente**, bloqueando a queima de prêmio pelo decaimento temporal.

4.4. Filtro de Salvaguarda Temporal: Janelas Parametrizadas e PDS

Próximo ao vencimento, os tokens sofrem severa perda de liquidez. O sistema adota a janela de corte padrão de 7 dias como parâmetro dinâmico (DAYS_TO_MATURITY_CUTOFF). Se DaysToMaturity < Cutoff, novas operações em PT entram em bloqueio total de salvaguarda (POOL_EXPIRING_SOON).

5. Engenharia Preditiva: Camada Inteligente e Fallback Determinístico

O módulo preditivo baseado em Machine Learning atua estritamente como um modificador de viés (*bias multiplier*) sobre as regras matemáticas determinísticas, prevenindo falhas catastróficas por sobreajuste (*overfitting*).

5.1. Mecanismo de Circuit Breaker de Confiança

Se a certeza do classificador estiver abaixo do limiar de segurança configurado de 65% (PREDICTOR_CONFIDENCE_THRESHOLD = 0.65), o módulo preditivo sofre bypass automático. O robô ignora a recomendação da inteligência e opera sob o Modo de Fallback Puramente Determinístico baseado em regras.

5.2. Interação com a Lógica de Decisão

  • Regime HIGH_YIELD (Bullish): Reduz o spread exigido aplicando um multiplicador de deságio na inércia (Δtarget_effective = Δtarget_effective × 0.85), facilitando a rotação rumo ao YT.
  • Regime LOW_YIELD (Bearish): Alarga o spread exigido (Δtarget_effective = Δtarget_effective × 1.25), inibindo aquisições de YT e direcionando a preferência para a imunização estável do PT.

5.3. Código Fonte de Produção do Módulo Preditivo

# market_regime_predictor.py
import numpy as np
import pandas as pd
from sklearn.cluster import KMeans
from sklearn.ensemble import RandomForestClassifier
from sklearn.preprocessing import StandardScaler
from sklearn.metrics import accuracy_score
from typing import Dict

class PendleMarketRegimePredictor:
    def __init__(self, n_regimes: int = 3, lookback_days: int = 30):
        self.n_regimes = n_regimes
        self.lookback = lookback_days
        self.kmeans = KMeans(n_clusters=n_regimes, random_state=42)
        self.classifier = RandomForestClassifier(
            n_estimators=200, max_depth=10, min_samples_split=5, random_state=42
        )
        self.scaler = StandardScaler()
        self.regime_labels = {0: "HIGH_YIELD", 1: "STABLE", 2: "LOW_YIELD"}

    def _extract_regime_features(self, df: pd.DataFrame) -> pd.DataFrame:
        features = pd.DataFrame(index=df.index)
        for window in [7, 14, 30]:
            features[f'apy_ma_{window}d'] = df['underlying_apy'].rolling(window).mean()
            features[f'apy_std_{window}d'] = df['underlying_apy'].rolling(window).std()
            features[f'tvl_ma_{window}d'] = df['tvl_usd'].rolling(window).mean()
        for period in [3, 7]:
            features[f'apy_momentum_{period}d'] = df['underlying_apy'].pct_change(period)
            features[f'spread_momentum_{period}d'] = (df['underlying_apy'] - df['implied_apy']).pct_change(period)
        
        features['apy_spread'] = df['underlying_apy'] - df['implied_apy']
        features['spread_ma_7d'] = features['apy_spread'].rolling(7).mean()
        features['spread_zscore'] = (features['apy_spread'] - features['apy_spread'].rolling(30).mean()) / features['apy_spread'].rolling(30).std()
        features['implied_lag'] = df['implied_apy'].shift(1) - df['implied_apy']
        features['convergence_rate'] = -features['implied_lag'] / (features['apy_spread'] + 0.001)
        features['tvl_growth_7d'] = df['tvl_usd'].pct_change(7)
        features['tvl_volatility'] = df['tvl_usd'].rolling(14).std() / df['tvl_usd'].rolling(14).mean()
        features['days_to_maturity'] = df['days_to_maturity']
        features['maturity_proximity'] = 1 / (features['days_to_maturity'] + 1)
        return features.dropna()

    def _label_regimes(self, df: pd.DataFrame, features: pd.DataFrame) -> pd.Series:
        clustering_features = features[['apy_ma_14d', 'apy_std_14d', 'spread_ma_7d', 'tvl_growth_7d', 'convergence_rate']]
        scaled = self.scaler.fit_transform(clustering_features)
        regimes = self.kmeans.fit_predict(scaled)
        regime_means = [features.loc[regimes == r, 'apy_ma_14d'].mean() for r in range(self.n_regimes)]
        order = np.argsort(regime_means)[::-1]
        regime_mapping = {old: new for new, old in enumerate(order)}
        return pd.Series(np.array([regime_mapping[r] for r in regimes]), index=features.index)

    def fit(self, historical_data: pd.DataFrame):
        features = self._extract_regime_features(historical_data)
        self.regime_labels_historical = self._label_regimes(historical_data, features)
        X, y = [], []
        for i in range(len(features) - 7):
            X.append(features.iloc[i].values)
            y.append(self.regime_labels_historical.iloc[i + 7])
        self.classifier.fit(np.array(X), np.array(y))
        return accuracy_score(np.array(y), self.classifier.predict(np.array(X)))

    def predict_regime(self, recent_data: pd.DataFrame, horizon_days: int = 7) -> Dict:
        features = self._extract_regime_features(recent_data)
        if len(features) == 0:
            return {'predicted_regime': 'UNKNOWN', 'confidence': 0.0, 'action': {'type': 'HOLD', 'multiplier': 1.0}}
        latest_features = features.iloc[-1:].values
        regime_id = self.classifier.predict(latest_features)[0]
        regime_probs = self.classifier.predict_proba(latest_features)[0]
        confidence = float(np.max(regime_probs))
        
        actions = {
            0: {'type': 'ROTATE_TO_YT', 'multiplier': 0.85},
            1: {'type': 'MAINTAIN_PT', 'multiplier': 1.0},
            2: {'type': 'SEEK_OPPORTUNITIES', 'multiplier': 1.25}
        }
        return {
            'predicted_regime': self.regime_labels[regime_id],
            'confidence': confidence,
            'regime_probabilities': {"HIGH_YIELD": regime_probs[0], "STABLE": regime_probs[1], "LOW_YIELD": regime_probs[2]},
            'action': actions[regime_id]
        }

6. Infraestrutura de Proteção, Matriz de Cooldown e Circuit Breaker Global

Para mitigar falhas sequenciais decorrentes de instabilidades temporárias de rede ou do nó de RPC, o agente calcula dinamicamente o parâmetro minOut com buffers de volatilidade coletados dos últimos 10 blocos na blockchain, aplicando uma matriz estrita de cooldown por categoria de erro:

Identificador do Erro Causa Raiz Comum Janela Isolada (Cooldown) Ação de Tratamento Corretivo
SLIPPAGE_VIOLATION Volatilidade repentina na AMM do Pendle 45 Segundos Recalcula volatility_bps e o buffer do roteador.
RPC_NODE_TIMEOUT Latência excessiva do provedor de infraestrutura 180 Segundos Efetua failover para nó alternativo da rede.
INSUFFICIENT_LIQUIDITY Tamanho do lote superior à profundidade do pool 900 Segundos Fraciona a ordem de tamanho ou aborta execução.
ORACLE_DISCREPANCY Desvio severo entre preço de tela e oráculo local 3600 Segundos Gela o mercado e aciona alarmes de verificação.
6.1. Circuit Breaker Global de Infraestrutura (Anti-Gas-Draining)

O sistema mantém um rastreador horário cumulativo (hourly_failure_count). Caso ocorram mais do que 5 falhas consecutivas dentro de uma janela deslizante de 60 minutos, o robô entra em estado CRITICAL_HALT: todas as operações automatizadas de escrita em carteira são paralisadas por completo e um alerta de emergência é disparado para intervenção manual imediata via canais de monitoramento (PagerDuty / Webhooks).
